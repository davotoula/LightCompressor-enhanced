# Release Design: Library `v1.9.0` + App `app-v1.4.0`

**Date:** 2026-04-11
**Base branch:** `master`
**Starting state:** clean working tree; library `1.8.4-SNAPSHOT`, app `1.3.3-SNAPSHOT` / `versionCode=133`
**Last releases:** `v1.8.3` (library, 2026-04-10), `app-v1.3.2` (app, 2026-04-10)

## Purpose

Ship a minor release of both components together. New user-visible feature since the last release is GIF → MP4 conversion (`GifToMp4Converter`) in the library, wired into the app's share/picker flows. This warrants a **minor** bump on both components per semver. Distribution targets: JitPack (library), GitHub Release + Google Play (AAB, manual upload) + Zapstore (APK, via `zsp`) for the app.

## Target versions

| Component | File | From | Release | Post-release |
|---|---|---|---|---|
| Library | `lightcompressor/gradle.properties` (`version=`) | `1.8.4-SNAPSHOT` | `1.9.0` | `1.9.1-SNAPSHOT` |
| App `appVersionName` | `gradle.properties` (root) | `1.3.3-SNAPSHOT` | `1.4.0` | `1.4.1-SNAPSHOT` |
| App `appVersionCode` | `gradle.properties` (root) | `133` | `140` | `141` |

`versionCode=140` is chosen to align numerically with `1.4.0` (matching the `app-v1.3.0` / `130` precedent). It is still monotonically increasing from `133`.

## Release phases

### Phase 1 — Library `v1.9.0` (autonomous)

1. Edit `lightcompressor/gradle.properties`: `version=1.9.0`
2. Commit: `Release commit: v1.9.0`
3. Tag: `git tag v1.9.0`
4. Push: `git push origin master v1.9.0`
5. Wait for `library-release.yml` to go green (watch via `gh run watch`)
6. Verify JitPack coordinate resolves (curl snippet from `RELEASE.md:126-139`):
   ```bash
   VERSION=1.9.0
   curl -sS -o /dev/null -w "POM %{http_code}\n" \
     "https://jitpack.io/com/github/davotoula/LightCompressor-enhanced/${VERSION}/LightCompressor-enhanced-${VERSION}.pom"
   curl -sS -o /dev/null -w "AAR %{http_code} %{size_download} bytes\n" \
     "https://jitpack.io/com/github/davotoula/LightCompressor-enhanced/${VERSION}/LightCompressor-enhanced-${VERSION}.aar"
   ```
   Both must return `200`. First call triggers JitPack's build (1–3 min); retry if 404.
7. Replace auto-notes with curated notes:
   ```bash
   gh release edit v1.9.0 --repo davotoula/LightCompressor-enhanced \
     --notes-file /tmp/lib-release-notes.md
   ```

### Phase 2 — App `app-v1.4.0` (autonomous + local verification pause)

After Phase 1, `lightcompressor/gradle.properties` is already `version=1.9.0` (no `-SNAPSHOT`). We do **not** bump it to `1.9.1-SNAPSHOT` between the two releases — doing both back-to-back avoids the temp-pin-then-revert dance from the `app-v1.3.2` precedent. The bump to `1.9.1-SNAPSHOT` happens once in Phase 4.

8. Edit root `gradle.properties`: `appVersionName=1.4.0`, `appVersionCode=140`
9. Edit `CHANGELOG.md` to add the `[app-v1.4.0]` section and update the footer comparison link
10. Commit: `Release commit: app-v1.4.0` (this commit touches `gradle.properties` and `CHANGELOG.md` only; library is untouched)
11. **Local verification (human-in-the-loop):**
    ```bash
    ./gradlew :app:assembleRelease
    adb install -r app/build/outputs/apk/release/lce-app-release.apk
    ```
    Open the app and visually confirm the bottom label at `MainScreen.kt:337` shows **"App v1.4.0 • Lib v1.9.0"** with no `-SNAPSHOT` suffix. If the label is wrong, abort and fix.
12. Tag: `git tag app-v1.4.0`
13. Push: `git push origin master app-v1.4.0`
14. Wait for `app-release.yml` green. The guard step will pass because `lightcompressor/gradle.properties` on the tagged commit is `1.9.0` (no `-SNAPSHOT`).

### Phase 3 — Distribution (stop & hand off)

15. **Google Play (manual):** I fetch the signed AAB URL:
    ```bash
    gh release view app-v1.4.0 --repo davotoula/LightCompressor-enhanced \
      --json assets --jq '.assets[] | select(.name | endswith(".aab")) | .url'
    ```
    Hand the URL to the user. The user downloads `lce-app-release.aab` and uploads it to Google Play Console manually.

16. **Zapstore (manual, needs nsec):** Hand the user this block to run locally from the repo root on the tagged commit:
    ```bash
    set -a
    source ~/.keys/lce/zapstore.env
    set +a
    zsp publish --check zapstore.yaml           # dry-run first
    zsp publish --skip-preview zapstore.yaml    # real publish
    ```
    `zapstore.yaml` already points to `release_notes: CHANGELOG.md`, so the new `[app-v1.4.0]` section becomes the Zapstore release description automatically.

### Phase 4 — Post-release bumps (autonomous)

17. Edit `lightcompressor/gradle.properties`: `version=1.9.1-SNAPSHOT`
18. Edit root `gradle.properties`: `appVersionName=1.4.1-SNAPSHOT`, `appVersionCode=141`
19. Commit: `Post release commit: app-v1.4.1-SNAPSHOT` (single commit covers both, per `6a39dda` precedent)
20. Push to master

## Risks and mitigations

| Risk | Mitigation |
|---|---|
| App ships showing `-SNAPSHOT` in the label | Phase 2 step 12 local visual check on device; CI guard in `app-release.yml` as second line |
| JitPack doesn't serve the tag | Phase 1 step 6 curl check before we tag the app |
| `app-release.yml` fails on the library guard | Phase 2 step 8 resets library to pinned `1.9.0` before the app tag |
| Zapstore publish fails (nsec missing, network) | Zapstore is the last step; GitHub release is already live so re-running is safe |
| Play upload rejected | User-side manual step; not automated. `versionCode=140 > 133` so monotonic rule is satisfied |
| Someone else pushes to master mid-release | `git status` clean before every push; I'll `git fetch && git status` before each tag push |

## Scope boundaries (what is NOT in this release)

- `todo.md` open items (GIF icon, remove pending videos, dev Firebase project) — separate work
- No changes to `google-services.json`, signing config, or CI workflow YAML
- No library API changes beyond what already landed on master since `v1.8.3`

## Rollback

- **Before tag push:** `git reset --hard origin/master` (uncommitted tag/commits are local-only)
- **After library tag, before app tag:** library release is public on JitPack; app release did not happen. To back out the library: `gh release delete v1.9.0 --yes && git push --delete origin v1.9.0`, then revert the library version commit. JitPack cached artifacts remain but are harmless.
- **After app tag:** do not roll back; ship a `1.4.1` patch instead.
